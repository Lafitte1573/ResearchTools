# VinciCoder: Unifying Multimodal Code Generation via Coarse-to-fine Visual Reinforcement Learning

## Core Ideas
**Research Problem:**
Current multimodal code generation models typically operate within narrow, task-specific paradigms (e.g., separate models for chart-to-code or web-to-HTML). These models rely heavily on Supervised Fine-Tuning (SFT) with textual instructions, which limits their ability to generalize across diverse visual domains. Furthermore, standard SFT lacks a mechanism to verify the visual fidelity of the generated code, as textual metrics (like token accuracy) do not correlate well with the visual correctness of the rendered output.

**Motivation:**
To develop a generalized "VisioN Code Intelligence," a unified model is needed that can handle various image-to-code tasks (charts, webpages, diagrams, scientific plots) without task-specific specialization. Additionally, a training strategy is required that optimizes for visual executability and fidelity rather than just textual similarity.

**Core Conceptual Contributions:**
1.  **VinciCoder:** A unified multimodal code generation model built on the QwenVL series.
2.  **Two-Stage Training Framework:** Combines large-scale SFT with a novel Visual Reinforcement Learning (ViRL) stage.
3.  **Visual-Based Code Refinement:** A novel task introduced during SFT where the model learns to rectify flawed code drafts to match a target image, enhancing robustness.
4.  **Coarse-to-fine Visual Reward:** A reinforcement learning reward mechanism that evaluates visual similarity at both global (coarse) and local (fine-grained) levels using visual embeddings.

## Key Innovations
*   **Unified Multimodal Architecture:** Unlike prior works that train separate models for specific domains (e.g., ChartCoder for charts, Web2Code for webpages), VinciCoder unifies five distinct domains (Chart-to-code, Web-to-HTML, Image-to-SVG, Image-to-LaTeX, and Scientific Plots) into a single model.
*   **Visual Reinforcement Learning (ViRL):** This is the first unified model to leverage Reinforcement Learning (RL) for domain-agnostic visual fidelity. It pivots the reward mechanism from brittle, rule-based textual validation to direct perceptual similarity signals.
*   **Coarse-to-fine Reward Mechanism:** To robustly compare high-resolution images, the ViRL framework employs a dual-granularity reward function:
    *   **Coarse-grained:** Evaluates global structure using downsampled thumbnails.
    *   **Fine-grained:** Evaluates local details by partitioning images into non-overlapping patches.
    *   This prevents the model from achieving high scores by merely matching global colors while ignoring structural details.
*   **Visual-Based Code Refinement Task (SFT):** Distinct from conventional instruction-based editing, this task conditions the model on a target image and a flawed code snippet (containing logical errors or partial renderings). The model must generate refined code that aligns visually with the target, building a robust foundation for the subsequent RL phase.

## Method & Implementation

### 1. Data Construction
The training data is constructed via a multi-stage pipeline to ensure diversity and quality.
*   **SFT Data (1.6M pairs):**
    *   **Sources:** Curated from open-source datasets including MCD, MSRL, Web2Code, UniSVG ISVGEN, ImgCode, and Cosyn-400k.
    *   **Tasks:** Covers Chart-to-code, Web-to-HTML, Image-to-SVG, Image-to-LaTeX, and Scientific Plots-to-code.
    *   **Refinement Dataset:** A novel subset where a specialized VLM generates initial (often flawed) code, which is paired with the ground-truth image and code. The model learns to map `(Image, Flawed Code) -> Refined Code`.
*   **RL Data (42k pairs):**
    *   Constructed to be mutually exclusive from SFT data.
    *   Focuses on high-fidelity, challenging samples across the five domains to ensure the RL phase drives meaningful improvement.
    *   Includes specific filtering (e.g., using Gemini-2.5-Flash to refine HTML code for complexity) to ensure quality.

### 2. Model Architecture
*   **Base Model:** Built upon the **QwenVL series** (specifically Qwen2.5-VL-7B/8B-Instruct).
*   **Input:** Accepts an Image and a Text Instruction.
*   **Output:** Generates executable code snippets (Python, HTML, SVG, LaTeX, etc.).

### 3. Training Strategy
The training follows a two-stage process:

#### Stage 1: Supervised Fine-Tuning (SFT)
*   **Objective:** Standard autoregressive loss to establish robust coding capabilities and learn fundamental templates.
*   **Formula:** $\mathcal{L}(\theta) := -\mathbb{E}_{(x,y) \sim \mathcal{D}_{\text{SFT}}} \sum_{t=1}^{T} \log P(y_t \mid x, y_{<t}; \theta)$
*   **Significance:** Provides the initial policy for the RL phase. The inclusion of the visual-based refinement task helps the model understand the relationship between code changes and visual outcomes.

#### Stage 2: Visual Reinforcement Learning (ViRL)
*   **Algorithm:** Uses **Group Relative Policy Optimization (GRPO)**. This algorithm samples a group of responses for a given query and optimizes the policy based on relative advantages, removing the need for a separate critic model.
*   **Reward Function:** The total reward $R$ is a combination of a Visual Reward ($R_v$) and a Language Alignment Reward ($R_l$).
    *   **Visual Reward ($R_v$):**
        1.  **Render:** The generated code is executed/rendered into an image ($I_r$).
        2.  **Embed:** Both the rendered image ($I_r$) and the ground-truth image ($I_s$) are encoded using **DINOv2-L** (a self-supervised ViT).
        3.  **Coarse-to-fine Comparison:**
            *   *Coarse:* A downsampled thumbnail of the entire image is compared to capture global layout.
            *   *Fine:* The image is partitioned into non-overlapping 448×448 grids. Local patches are compared to capture fine details.
            *   *Metric:* Normalized cosine similarity between DINOv2 embeddings.
            *   $R_v(I_s, I_r) = \frac{1}{2}(\cos(\text{DINO}(I_s), \text{DINO}(I_r)) + 1)$
    *   **Language Alignment Reward ($R_l$):**
        *   Ensures the generated code matches the requested language (e.g., if the prompt asks for Python, the output shouldn't be LaTeX).
        *   Computed by parsing the target language from the instruction and checking for valid aliases in the generated code.
        *   Binary reward (1 if match, 0 otherwise).
    *   **Final Reward:** $R = \omega_v \frac{1}{N} \sum R_{v,i} + \omega_l R_l$ (where $\omega_v=0.9, \omega_l=0.1$).

## Experimental Results

### 1. Experimental Setup
*   **Baselines:** Compared against closed-source models (Gemini-2.5-Pro, Claude-3.5-Sonnet, GPT-4o) and open-source models (InternVL series, Qwen2.5-VL series).
*   **Benchmarks:** Evaluation spans five domains:
    1.  **ChartMimic:** Chart-to-code (Python).
    2.  **UniSVG-ISVGEN:** Image-to-SVG.
    3.  **Design2Code:** Webpage-to-HTML.
    4.  **Image2Latex_plot:** Image-to-LaTeX.
    5.  **ChemDraw:** Molecule-to-SMILES/Code.
*   **Metrics:** Execution Rate, Low-Level/High-Level scores (SSIM, LPIPS), Render Success Rate, Earth Mover Similarity (EMS), and Tanimoto Similarity.

### 2. Main Findings
*   **State-of-the-Art Performance:** VinciCoder sets a new SOTA among open-source models across the vast majority of benchmarks.
*   **Comparison with Closed-Source:**
    *   VinciCoder-8B outperforms proprietary models like Gemini-2.5-Pro and Claude-3.5-Sonnet on specific tasks such as **Image-to-SVG** (UniSVG-ISVGEN) and **ChemDraw**.
    *   On ChartMimic, it remains competitive but closed-source models retain a slight edge in complex reasoning.
*   **Effectiveness of ViRL:**
    *   The ViRL stage provides significant gains over the SFT-only version. For example, on the UniSVG-ISVGEN High-L score, VinciCoder-8B improved from 87.6 (SFT) to **94.1** (ViRL).
    *   On Design2Code, the High-L score improved from 86.8 to **89.3**.
*   **Model Scaling:** The 8B parameter version consistently outperforms the 7B version, demonstrating effective scaling.

### 3. Specific Benchmark Highlights (VinciCoder-8B)
*   **ChartMimic (Direct):** Exec. Rate **91.6** (vs. Gemini-2.5-Pro 97.3).
*   **UniSVG-ISVGEN:** High-L Score **94.1** (vs. Gemini-2.5-Pro 80.3). *Significant lead.*
*   **Design2Code:** High-L Score **89.3** (vs. Claude-3.5-Sonnet 90.8). *Highly competitive.*
*   **ChemDraw:** Tani. Sim. **62.6** (vs. GPT-4o 52.1). *Surpasses closed-source.*

### 4. Ablation Studies
*   **Coarse-to-fine Reward:** Validated that combining coarse (global) and fine (local) rewards yields better visual fidelity than using either alone. The coarse reward ensures layout correctness, while the fine reward ensures detail accuracy.
*   **Refinement Task:** The inclusion of the visual-based code refinement task in SFT was shown to build a more robust foundation for the RL phase compared to standard direct generation data alone.

### 5. Implementation Details
*   **Hardware:** Training performed on 24 H800 GPUs.
*   **Batch Size:** Global batch size of 96 for SFT; 256 for RL.
*   **RL Configuration:** 16 GPUs for the policy model, 4 GPUs for reward scoring.
*   **Visual Encoder:** DINOv2-L utilized for visual embedding during the reward calculation phase.
