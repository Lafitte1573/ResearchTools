# VisCodex: Unified Multimodal Code Generation via Merging Vision and Coding Models

## Core Ideas
The paper addresses the limitation of current Multimodal Large Language Models (MLLMs), which excel at visual understanding but lack the deep programming knowledge required for robust code generation from visual inputs (e.g., translating a UI mockup to HTML or replicating a data chart). The core research problem is bridging the gap between visual perception and code generation without incurring the high costs of pre-training a new model from scratch.

**VisCodex** is proposed as a unified framework that seamlessly merges a state-of-the-art vision-language model (VLM) with a dedicated coding Large Language Model (LLM). The central hypothesis is that by arithmetically merging the parameters of these specialized models using **task vectors**, the resulting model can retain strong visual comprehension while acquiring advanced coding skills. To support this, the authors introduce the **Multimodal Coding Dataset (MCD)** for training and **InfiBench-V**, a new benchmark for evaluating visually-rich programming tasks.

## Key Innovations
*   **Task Vector-Based Model Merging for MLLMs:** Unlike standard fine-tuning, VisCodex employs a parameter merging technique. It extracts task vectors representing the parameter shifts from a base model to a specialized coding model and merges them into a VLM's language backbone. This allows for the integration of distinct skills (vision vs. coding) efficiently.
*   **Multimodal Coding Dataset (MCD):** A large-scale, high-quality instruction-tuning dataset comprising **598k samples**. It is curated from four diverse sources to cover a wide spectrum of coding tasks:
    1.  **Enhanced HTML Code:** Aesthetically improved webpages generated from screenshots.
    2.  **Chart Image-Code Pairs:** High-quality Python matplotlib code paired with charts.
    3.  **Image-Augmented Code QA:** Real-world StackOverflow questions with images.
    4.  **Algorithmic Code:** Foundational problems to preserve reasoning abilities.
*   **InfiBench-V Benchmark:** A novel evaluation benchmark specifically designed for **visually-rich, real-world programming questions**. Unlike text-only benchmarks, InfiBench-V requires the model to understand visual context (e.g., UI layouts, error screenshots) to formulate a correct solution. It spans 13 programming languages and 5 high-level categories (Front-end, Back-end, DS&ML, Mobile/Desktop, IT Ops).
*   **Selective Parameter Merging:** The merging process targets only the **language model backbone**, leaving the vision encoder and cross-modal projection modules intact. This preserves the intrinsic visual understanding capabilities of the base VLM while upgrading its generative logic.

## Method & Implementation

### Model Architecture
*   **Foundation Model:** The framework adopts **Qwen2.5-VL** as the base architecture. This choice is driven by its use of 2D Rotary Position Embedding (RoPE) within the Vision Transformer (ViT), allowing flexible processing of images with arbitrary resolutions.
*   **Components:** The MLLM consists of a vision encoder, a language model backbone, and a projection module connecting the two.

### Model Merging Strategy
The core mechanism involves linearly merging task vectors derived from fine-tuned specialized models.
1.  **Task Vector Definition:** A task vector $\tau_{task}$ quantifies the parameter shift resulting from fine-tuning a base model $\theta_{base}$ on a specific task:
    $$ \tau_{task} = \theta_{ft} - \theta_{base} $$
2.  **Specific Vectors:**
    *   **VLM Task Vector ($\tau_{vlm}$):** Captures shifts for handling multimodal inputs ($\theta_{vlm} - \theta_{base}$).
    *   **Code Task Vector ($\tau_{code}$):** Captures shifts for code comprehension and generation ($\theta_{code} - \theta_{base}$).
3.  **Merging Formula:** The updated language model parameters $\theta_{VisCodex}$ are computed as:
    $$ \theta_{VisCodex} = \theta_{base} + \lambda \tau_{vlm} + (1 - \lambda) \tau_{code} $$
    Where $\lambda \in [0, 1]$ controls the trade-off between retaining original multimodal representations and integrating new code expertise.

### Implementation Details
*   **Base Models:** Qwen2.5-VL (7B and 32B scales).
*   **Coding Models for Merging:**
    *   For the 7B scale: **OpenCodeReasoning-Nemotron-1.1-7B**.
    *   For the 32B scale: **OpenCodeReasoning-Nemotron-1.1-32B**.
*   **Training:** After merging, the model undergoes supervised fine-tuning (SFT) on the MCD dataset. During this phase, the vision encoder and projection modules are **frozen**, and only the language model backbone is fine-tuned to align the merged capabilities with multimodal coding tasks.

### Dataset Construction (MCD)
*   **Enhanced HTML (200k pairs):** Generated by using GPT-4o to redesign existing webpages from Web2Code seeds, rendered via Playwright, and filtered for visual artifacts.
*   **Chart Image-Code (210k pairs):** Combines 164k synthetic samples (Chart2Code) with 46k real-world samples curated from GitHub. Real-world scripts were rewritten and normalized using GPT-4o to ensure executability and quality.
*   **Image-Augmented QA (59k pairs):** Sourced from StackOverflow. A rigorous pipeline removed low-quality entries, and GPT-4o was used to refine answers for clarity and conciseness.
*   **Algorithmic Code (129k pairs):** Aggregated from established datasets like LeetCode, Codeforces, and TACO to maintain core reasoning skills.

## Experimental Results

### Evaluation Benchmarks
The model is evaluated on four multimodal benchmarks to assess a range of coding skills:
1.  **Design2Code:** Measures the ability to translate visual UI designs into executable code (evaluated on Low-Level features and High-Level semantic fidelity).
2.  **ChartMimic:** Evaluates the generation of chart specifications from images (Direct Mimic task).
3.  **MMCode:** Assesses algorithmic problem-solving in visually rich contexts (measured by pass@1 accuracy).
4.  **InfiBench-V (Ours):** The proposed benchmark for complex, image-dependent programming questions.

### Evaluation Criteria for InfiBench-V
To ensure robust scoring on the new benchmark, a three-pronged strategy is adopted:
*   **Keyword Matching:** Checks for the presence of specific essential terms and phrases defined by domain experts.
*   **Unit Testing:** For code-heavy questions, correctness is verified via automated unit tests with precise I/O formats.
*   **GPT-4o Judge:** For questions relying on natural language understanding, GPT-4o scores responses based on correctness and completeness compared to a reference answer.

### Performance Claims
*   **State-of-the-Art:** Extensive experiments indicate that VisCodex achieves state-of-the-art performance among **open-source MLLMs**.
*   **Competitive with Proprietary Models:** The merged model approaches the performance of proprietary models like **GPT-4o** on multimodal coding tasks.
*   **Effectiveness of Merging:** The merged model significantly outperforms the original vision-language model on multimodal coding tasks, validating the efficiency of the task vector merging strategy.
*   **Data Impact:** The use of the MCD dataset and the specific merging of coding task vectors are highlighted as key factors in bridging the gap between visual perception and coding proficiency.
