# VIS-Shepherd: Constructing Critic for LLM-based Data Visualization Generation

## Core Ideas
**Research Problem:**
While Large Language Models (LLMs) have shown promise in automating data visualization generation from natural language instructions, they frequently produce suboptimal or deficient visualizations that require human intervention. Existing critique models are often text-based or lack the specialized domain knowledge required to evaluate rendered visual representations effectively. There is a significant gap in automated mechanisms that can systematically identify visual defects and provide constructive feedback for LLM-generated charts.

**Motivation:**
Visualization assessment fundamentally differs from text or code generation because the final output is a rendered image requiring visual perception and domain-specific design principles to evaluate. Current methods fall short because they rely solely on textual input or lack the empirical understanding of visualization quality. There is a need for a specialized "critic" that can look at the generated chart, compare it against the intent, and offer actionable improvements.

**Core Conceptual Contributions:**
*   **VIS-Shepherd:** A specialized Multimodal Large Language Model (MLLM) designed specifically to evaluate and provide feedback on LLM-generated data visualizations.
*   **Critique-Driven Framework:** Instead of just generating visualizations, the system introduces a feedback loop where the MLLM acts as a shepherd, guiding the generation process by identifying defects (e.g., visual clarity, instruction compliance) and suggesting improvements.
*   **High-Quality Critique Dataset:** A systematic framework to construct a dataset of visualization critiques, moving from raw human-created visualizations to LLM-generated counterparts paired with expert human feedback.

## Key Innovations
*   **Specialized MLLM Critic:** Unlike general-purpose critics, VIS-Shepherd is fine-tuned specifically on visualization defects. It processes the instruction, data, and the rendered visualization image to output structured feedback.
*   **Systematic Dataset Construction Pipeline:** The authors propose a novel four-stage framework to create training data where none existed:
    1.  **Curation:** Filtering real-world human visualizations.
    2.  **Synthesis:** Generating instructions and exporting data from those visualizations.
    3.  **Generation:** Using SOTA LLMs to create "flawed" versions of the human visualizations.
    4.  **Annotation:** Having experts critique the LLM versions against the human "gold standard."
*   **Defect Categorization Taxonomy:** The work establishes a structured categorization of common visualization defects (Instruction Compliance, Visual Clarity, Semantical Readability) to guide annotation and model training, distinguishing between disqualifying defects and preferential suggestions.
*   **Efficiency via Small Models:** The work demonstrates that a relatively small (7B parameter) open-source MLLM, when trained on high-quality, domain-specific critique data, can achieve performance comparable to significantly larger proprietary models.

## Method & Implementation

### 1. VIS-Shepherd Framework Overview
The system operates in an end-to-end pipeline:
1.  **Input:** User provides an instruction and a dataset.
2.  **Generation:** An LLM generates visualization code (e.g., Python/D3), which is executed to produce a rendered image.
3.  **Critique:** VIS-Shepherd takes the instruction, data preview, and rendered image as input.
4.  **Output:** The model outputs a critique identifying specific defects and offering suggestions for improvement.

### 2. Dataset Construction Framework
To train VIS-Shepherd, the authors constructed a high-quality dataset through a four-stage process:

**Stage 1: Human-created Instance Curation**
*   **Source:** Crawled 180,000 real-world human-created visualization instances with code from **Observable** (a public platform).
*   **Filtering:**
    *   Used a closed-source MLLM to filter out poor quality (blank outputs, incomplete representations) or interactive/toy datasets.
    *   Result: 67K instances.
    *   **Manual Inspection:** Human annotators with visualization expertise inspected and selected high-quality instances.
    *   **Deduplication:** Used simhash and manual checks on rendered outputs.
    *   **Final Output:** A curated set of **1.7K diverse, high-quality human-created visualization exemplars**.

**Stage 2: Instruction Synthesis and Dataset Exportation**
*   **Goal:** Create triplets $(I, D, V_h)$ where $I$=Instruction, $D$=Dataset, $V_h$=Human Visualization.
*   **Instruction Synthesis:** Employed a role-playing approach where an LLM imagined user backgrounds and usage scenarios for each human visualization to generate plausible natural language instructions.
*   **Dataset Exportation:** Since original code often hard-codes data, an LLM rewrote the visualization code to export data primarily in **CSV format** (or GeoJSON for maps) to ensure standardization.
*   **Validation:** Ensured the rewritten code produced the same visual effect as the original.

**Stage 3: LLM-based Visualization Generation**
*   **Goal:** Create quadruples $(I, D, V_h, V_{LLM})$.
*   **Process:** Used advanced models (**GPT-4o** and **Claude-Sonnet-3.5**) to generate visualizations ($V_{LLM}$) based on the synthesized instructions ($I$) and exported data ($D$).
*   **Library:** Directed LLMs to utilize **D3**, a widely adopted imperative visualization library, to maximize expressiveness.
*   **Filtering:** Filtered out instances where code triggered compiler errors (as these are handled by compilers, not visual critics).

**Stage 4: High Quality Critique Collection**
*   **Goal:** Create quintuplets $(I, D, V_h, V_{LLM}, C)$ where $C$=Critique.
*   **Critique Categories:** Defined two main cases:
    *   **Case 1:** Identifying defects that disqualify the visualization (needs correction).
    *   **Case 2:** Visualizations without disqualifying defects, offering preferential suggestions (aesthetics, encoding effectiveness).
*   **Annotation Process:**
    *   **Preliminary:** 3 visualization domain experts analyzed 200 samples to define defect categories (see Table 1 in paper).
    *   **Main Annotation:** 10 annotators with visualization expertise compared $V_{LLM}$ against $V_h$.
    *   **Assistance:** Used GPT-4o to automatically compare $V_{LLM}$ and $V_h$ to generate potential preferential suggestions for annotators to reference.
    *   **Quality Assurance:** Periodic sampling of annotator work.
*   **Final Dataset:** **2.7K high-quality critique annotations** (2.5K for training, 0.16K for testing).

### 3. Model Training Configuration
*   **Base Model:** **Qwen-2.5-VL-7B** (an open-source Multimodal LLM).
*   **Optimizer:** AdamW with learning rate $1e-5$, $\beta_1=0.9$, $\beta_2=0.999$, $\epsilon=1e-8$, weight decay 0.01.
*   **Scheduler:** Cosine learning rate scheduler.
*   **Training Duration:** 1 epoch (to prevent overfitting on the small dataset).
*   **Batch Size:** 8 with gradient accumulation step of 1.
*   **Sequence Length:** Maximum 4096 tokens.
*   **Hardware:** 8 NVIDIA A800 (80GB) GPUs.
*   **Precision:** bf16.
*   **Inference:** Temperature fixed to 0 for confident predictions.

### 4. Defect Taxonomy (Table 1)
The annotation was guided by specific defect types:
*   **Instruction Compliance:** Failing to meet requirements specified in user instructions (e.g., wrong chart type, missing data).
*   **Visual Clarity:** Issues like overlapping elements, undersized text, or crowding that impair readability.
*   **Semantical Readability:** Unclear meanings, confusing legends, or defective labeling.
*   **No Defect:** Cases where the visualization is valid, but preferential suggestions can still improve hierarchy or spacing.

## Experimental Results

**Dataset Statistics:**
*   **Initial Crawl:** 180K instances from Observable.
*   **Curated Human Viz:** 1.7K high-quality instances.
*   **Final Critique Dataset:** 2.7K high-quality annotations.

**Model Performance:**
*   **Parameter Efficiency:** The fine-tuned **7B parameter model** achieved substantial performance gains.
*   **Competitiveness:** Despite its compact size, VIS-Shepherd attained performance levels comparable to significantly larger (e.g., 10x size) state-of-the-art open-source and proprietary models.
*   **Evaluation Methods:**
    *   **Model-based Automatic Evaluation:** Used to assess the quality of the generated critiques.
    *   **Human Preference Studies:** Conducted to validate the effectiveness and helpfulness of the feedback provided by VIS-Shepherd.

**Significance:**
The experiments validate that high-quality, domain-specific data (the 2.7K critiques) is more critical than model scale for this specific task. The work demonstrates significant potential for MLLM-based automated visualization critique, reducing the need for human intervention in the visualization generation pipeline.
